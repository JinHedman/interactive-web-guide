You have a published branch with a couple of focused commits. Now you need to bring that work back into `main` — but not blindly. A **pull request** (PR) is the moment where work is proposed, looked over, discussed, and only then merged. This chapter takes your branch from "published" to "merged into `main`," then back down to your computer.

## What a pull request is for

A pull request says: *"Here are my commits on this branch. I want to merge them into `main`. Please review."* It is not just a merge button — it is a place for conversation.

A PR gives you:

- **A clear diff.** GitHub shows exactly which lines were added and removed, so a reviewer can read the change without running anything.
- **Discussion and review.** Teammates can comment on specific lines, ask questions, request changes, or approve. On a solo project you are your own reviewer — reading your own diff catches surprising numbers of mistakes.
- **A record.** Every PR is a titled, described, linked history of why a change was made, kept long after the branch is gone.

Even working alone, PRs are worth the habit: they force a last look before anything touches `main`.

## Open a pull request

Right after you publish a branch, GitHub Desktop offers to start a PR from it.

1. With your branch selected as the Current Branch, click **Preview Pull Request**. GitHub Desktop shows the diff between your branch and `main` so you can sanity-check what you are about to propose.
2. If it looks right, click **Create Pull Request**. (You can also use the dropdown next to the button to go straight to **Create Pull Request** without the preview.)
3. GitHub Desktop opens your browser on github.com, on the "Open a pull request" page, with your branch already set to merge into `main`.
4. On github.com, type a **title** and a short **description** of what the change does and why, then click **Create pull request**.

The PR now exists on GitHub. Underneath, it is simply a request to merge your branch's commits into the base branch — but wrapped in a page built for review.

## Review the changes

On the PR page, click the **Files changed** tab. This is the diff: green lines were added, red lines were removed. Read it as if you were a teammate seeing the work for the first time. Ask:

- Does every change belong to this branch's one job?
- Did anything sneak in by accident — a stray edit, a debugging line, a misspelling?
- Is the result something you would be happy to find on `main`?

On a team, a reviewer leaves comments here and clicks **Review changes** to approve or request changes. Solo, you do the same read yourself. If you spot a problem, you do not abandon the PR — you switch back to the branch in GitHub Desktop, commit a fix, and push. The PR updates automatically, because it always reflects the branch's latest commits.

## Merge the pull request

When the change is approved and GitHub reports no conflicts, click the green **Merge pull request** button on the PR's **Conversation** tab, then **Confirm merge**. GitHub combines your branch's commits into `main`. After merging, GitHub offers a **Delete branch** button — once a branch is merged, deleting it keeps the branch list clean. The work lives on in `main`; the branch has done its job.

## Pull the merged result back

Here is the step people forget. The merge happened on GitHub, in the cloud. Your computer's `main` does not know about it yet.

1. In GitHub Desktop, switch the **Current Branch** dropdown back to `main`.
2. Click **Fetch origin** to check GitHub for new commits, then click **Pull origin** to download them.

**Pulling** brings GitHub's commits down to your computer — the mirror image of pushing. After pulling, your local `main` contains the merged work, and you are ready to branch off it again for the next task. That cycle — branch, commit, publish, pull request, merge, pull `main` — is the everyday rhythm of working with Git.
