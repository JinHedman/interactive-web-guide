Your profile card looks fine on a laptop, but phones, tablets, and large monitors all have different widths. **Responsive** design means one page that adapts to whatever screen it lands on. This closing chapter covers the three tools that make it happen: the viewport meta tag, fluid sizing, and media queries.

## The viewport meta tag

Without help, mobile browsers pretend the screen is about 980px wide and shrink the whole page to fit — text becomes tiny. The viewport meta tag tells the browser to use the device's real width and not zoom out. Put it in the `<head>` of every page (you have seen it since the HTML module):

<CodeExample language="html">
{`<meta name="viewport" content="width=device-width, initial-scale=1">`}
</CodeExample>

`width=device-width` matches the layout to the device's width; `initial-scale=1` sets the starting zoom. Without this tag, no amount of CSS will make a page feel right on a phone.

## Fluid sizing first

Before reaching for media queries, let the layout flex on its own. Two habits do most of the work:

- Use **relative units** (`rem`, `%`) rather than fixed pixels for sizes that should adapt.
- Use **`max-width`** instead of `width` so a box can shrink on small screens but stops growing on large ones.

<CodeExample language="css">
{`.card {
  width: 100%;        /* fill a narrow screen */
  max-width: 360px;   /* but never grow past 360px */
  margin: 0 auto;     /* center it once there is spare room */
  padding: 1.5rem;
}`}
</CodeExample>

That card is already responsive: full width on a phone, capped and centered on a desktop — with no media query at all. Reach for fluid sizing first; add breakpoints only where the layout genuinely needs to change.

## Media queries

A media query applies a block of CSS only when a condition is true — usually a screen-width range. You write the base styles for small screens, then add a query that adjusts them once there is more room. This **mobile-first** approach keeps the simplest layout as the default.

<CodeExample language="css">
{`/* base: mobile — stack vertically */
.card {
  display: flex;
  flex-direction: column;
  gap: 1rem;
}

/* 600px and up — switch to a row */
@media (min-width: 600px) {
  .card {
    flex-direction: row;
    align-items: center;
  }
}`}
</CodeExample>

The width at which the layout changes — here `600px` — is called a **breakpoint**. Choose breakpoints where *your* design starts to look cramped, not by chasing specific device sizes.

<ReadMore title="How teams write CSS today: Tailwind CSS, CSS-in-JS, and Sass">
You have written CSS the classic way — rules in a stylesheet, targeting classes. Many teams now use **Tailwind CSS**, a *utility-first* framework. Instead of writing a rule, you compose a design from tiny single-purpose classes applied directly in your markup — which reads **a bit like inline styles**, except the values come from a consistent design system and you can still use hover states, breakpoints, and dark mode. Compare the two approaches for our card heading:

<CodeExample language="html">
{`<!-- Plain CSS: a rule in your stylesheet -->
<style>
  .title { font-size: 1.25rem; font-weight: 700; color: navy; }
</style>
<h1 class="title">Ada Lovelace</h1>

<!-- Tailwind: utility classes right on the element -->
<h1 class="text-xl font-bold text-blue-900">Ada Lovelace</h1>`}
</CodeExample>

Each Tailwind class maps to one declaration: `text-xl` is a font size, `font-bold` is `font-weight: 700`, `text-blue-900` is a color. Tailwind's current major version is **v4**, which you set up with a single `@import "tailwindcss";` line in your CSS — no large config file required. Two other approaches you will meet: **CSS-in-JS** (libraries like styled-components that let you write CSS inside JavaScript components), and **Sass**, an older preprocessor that adds variables, nesting, and reusable mixins to plain CSS. Skim the links below when you are curious; the box-model, selector, flexbox, and media-query foundations from this module carry into all of them.

- [Tailwind CSS — official docs](https://tailwindcss.com/docs)
- [CSS grid layout (MDN)](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_grid_layout)
- [Sass — official site](https://sass-lang.com/)
</ReadMore>
