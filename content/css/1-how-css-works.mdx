Your HTML page already works, but it looks plain — black text, blue links, default fonts. CSS (Cascading Style Sheets) is the language that controls how that markup *looks*: colors, spacing, fonts, and layout. In this module you will progressively style a personal profile card. This first chapter covers the shape of a CSS rule, the three kinds of selectors you will use most, and where to put your styles.

## A rule is a selector plus declarations

Every CSS rule has two parts: a **selector** that picks which elements to style, and a **declaration block** in `{ }` that lists what to change. Each declaration is a `property: value` pair ending in a semicolon.

<CodeExample language="css">
{`h1 {
  color: navy;
  font-size: 32px;
}`}
</CodeExample>

Read that as: "find every `<h1>`, then set its `color` to `navy` and its `font-size` to `32px`." `h1` is the selector; `color` and `font-size` are properties; `navy` and `32px` are their values.

## Element, class, and id selectors

You have three everyday ways to target elements.

- **Element selector** — matches every tag of that type: `p`, `h1`, `a`.
- **Class selector** — matches elements with a given `class` attribute. Write it with a leading dot: `.card`. Classes are reusable; many elements can share one.
- **Id selector** — matches the one element with a given `id`. Write it with a leading hash: `#intro`. An `id` must be unique on the page.

<CodeExample language="css">
{`/* every paragraph */
p {
  line-height: 1.5;
}

/* any element with class="card" */
.card {
  background: white;
}

/* the single element with id="intro" */
#intro {
  font-weight: bold;
}`}
</CodeExample>

Reach for classes by default. They are reusable and keep your styling flexible. Use element selectors for broad defaults, and ids sparingly.

## See selectors in action

Here is the start of our profile card, split the way real projects keep it: an `index.html` for the markup and a `styles.css` for the rules. The HTML carries a `class` and an `id`, and links the stylesheet from the `<head>`; the stylesheet targets each selector.

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <link rel="stylesheet" href="styles.css">
  </head>
  <body>
    <div class="card">
      <h1 id="name">Ada Lovelace</h1>
      <p>I find patterns in numbers and write algorithms for fun.</p>
    </div>
  </body>
</html>`
  },
  {
    name: "styles.css",
    lang: "css",
    code: `body { font-family: system-ui, sans-serif; padding: 1.5rem; }
.card { background: #f3f4f6; padding: 1rem; }
#name { color: navy; }
p { color: #444; }`
  }
]} />

The `<link>` connects the two files, so the rules in `styles.css` apply to the elements in `index.html`. We will keep our styles in a separate file like this from now on.

## Three places CSS can live

CSS can sit in three places, from least to most maintainable.

**Inline** — a `style` attribute on a single element. Quick, but it styles only that one element and is hard to reuse or override.

<CodeExample language="html">
{`<h1 style="color: navy; font-size: 32px;">Ada Lovelace</h1>`}
</CodeExample>

**Internal** — a `<style>` block in the `<head>`. Fine for a quick demo or a one-page site, but the styles cannot be shared across pages.

<CodeExample language="html">
{`<head>
  <style>
    h1 { color: navy; }
  </style>
</head>`}
</CodeExample>

**External** — a separate `.css` file linked from the `<head>`. This is the one to prefer: one stylesheet can style every page, and your HTML stays clean. It is two files working together — `index.html` links the stylesheet with a `<link>` tag, and `styles.css` holds the rules:

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <link rel="stylesheet" href="styles.css">
  </head>
  <body>
    <h1>Ada Lovelace</h1>
  </body>
</html>`
  },
  {
    name: "styles.css",
    lang: "css",
    code: `h1 {
  color: navy;
  font-size: 32px;
}`
  }
]} />

The `<link>` tag's `href` points at the file; the matching `styles.css` holds your rules. This is the form to reach for, and the one the rest of this module uses.

## The cascade and specificity, briefly

What happens when two rules set the same property on the same element? CSS resolves the conflict with two ideas.

**The cascade**: when rules are equally specific, the one written *later* wins. Source order matters.

**Specificity**: a more specific selector beats a less specific one, regardless of order. The rough ranking is: id (`#name`) beats class (`.card`) beats element (`h1`). An inline `style` attribute is stronger still.

<CodeExample language="css">
{`p { color: gray; }        /* element: weak */
.card p { color: black; } /* class + element: stronger, wins */`}
</CodeExample>

You do not need to master specificity now. Just remember the order of strength, and that you can usually avoid conflicts by leaning on classes and keeping your stylesheet tidy.
