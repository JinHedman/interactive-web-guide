This is where it all comes together. You have selected elements, changed content and styles, listened for events, read form input, and created and removed elements. Now you will assemble those skills into one complete, working **to-do list app** — the project you have been building toward all module. Take your time; this chapter is a build, not a read.

## Plan before you code

A few minutes of planning saves an hour of confusion. Three questions:

1. **What is the data?** A list of tasks. Each task has text and a done flag — so an **array of objects**, like `{ text: "Buy milk", done: false }`.
2. **What can the user do?** Add a task, toggle a task complete, and delete a task.
3. **What renders?** A list where each task shows its text, looks crossed-out when done, and has a delete button.

The guiding rule, from the last chapter: **the array is the source of truth.** Every user action changes the array, then you call `render` to redraw the page from it. You never hand-edit the DOM and the data separately — that is how apps get out of sync.

## Build the structure and styles

Start with the HTML shell and a little CSS. A flex row keeps each task's text, checkbox, and delete button aligned. The `done` class provides the crossed-out look.

## Render the task array

Now the engine. Hold the tasks in an array. The `render` function clears the list, loops over the array, and builds one `<li>` per task — text in a `<span>`, plus a checkbox and a delete button. Re-running `render` always redraws from the current data.

<CodeExample language="html" preview>
{`<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <style>
      body { font-family: system-ui, sans-serif; max-width: 28rem; margin: 2rem auto; }
      ul { list-style: none; padding: 0; }
      li { display: flex; align-items: center; gap: 0.5rem; padding: 0.4rem 0; }
      li span { flex: 1; }
      .done span { text-decoration: line-through; color: gray; }
    </style>
  </head>
  <body>
    <h1>My Tasks</h1>
    <ul id="taskList"></ul>
    <script>
      const list = document.getElementById("taskList");

      const tasks = [
        { text: "Buy milk", done: false },
        { text: "Walk the dog", done: true },
      ];

      function render() {
        list.textContent = "";
        for (const task of tasks) {
          const li = document.createElement("li");
          if (task.done) {
            li.classList.add("done");
          }

          const span = document.createElement("span");
          span.textContent = task.text;

          li.append(span);
          list.append(li);
        }
      }

      render();
    </script>
  </body>
</html>`}
</CodeExample>

One starter task is already done, so it renders crossed out — proof the `done` flag drives the look. Next you add the actions.

## Add, complete, and remove

Here is the finished app. Read each handler as "change the array, then `render`":

- **Add** — the form's submit handler trims the input, pushes a new `{ text, done: false }` object, clears the input, and re-renders.
- **Complete** — each task's checkbox click flips `task.done`, then re-renders so the strike-through appears or disappears.
- **Remove** — each task's delete button splices that task out of the array, then re-renders.

<CodeExample language="html" preview>
{`<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <style>
      body { font-family: system-ui, sans-serif; max-width: 28rem; margin: 2rem auto; }
      form { display: flex; gap: 0.5rem; }
      input[type="text"] { flex: 1; padding: 0.4rem; }
      ul { list-style: none; padding: 0; }
      li { display: flex; align-items: center; gap: 0.5rem; padding: 0.4rem 0; }
      li span { flex: 1; }
      .done span { text-decoration: line-through; color: gray; }
      .empty { color: gray; font-style: italic; }
    </style>
  </head>
  <body>
    <h1>My Tasks</h1>
    <form id="taskForm">
      <input type="text" id="taskInput" placeholder="What needs doing?">
      <button type="submit">Add</button>
    </form>
    <ul id="taskList"></ul>

    <script>
      const form = document.getElementById("taskForm");
      const input = document.getElementById("taskInput");
      const list = document.getElementById("taskList");

      const tasks = [
        { text: "Buy milk", done: false },
        { text: "Walk the dog", done: true },
      ];

      function render() {
        list.textContent = "";

        if (tasks.length === 0) {
          const li = document.createElement("li");
          li.textContent = "No tasks yet. Add one above.";
          li.className = "empty";
          list.append(li);
          return;
        }

        tasks.forEach(function (task, index) {
          const li = document.createElement("li");
          if (task.done) {
            li.classList.add("done");
          }

          const checkbox = document.createElement("input");
          checkbox.type = "checkbox";
          checkbox.checked = task.done;
          checkbox.addEventListener("change", function () {
            tasks[index].done = checkbox.checked;
            render();
          });

          const span = document.createElement("span");
          span.textContent = task.text;

          const delBtn = document.createElement("button");
          delBtn.textContent = "Delete";
          delBtn.addEventListener("click", function () {
            tasks.splice(index, 1);
            render();
          });

          li.append(checkbox, span, delBtn);
          list.append(li);
        });
      }

      form.addEventListener("submit", function (event) {
        event.preventDefault();
        const text = input.value.trim();
        if (text === "") {
          return;
        }
        tasks.push({ text: text, done: false });
        input.value = "";
        render();
      });

      render();
    </script>
  </body>
</html>`}
</CodeExample>

Try the whole thing in the preview: add a task, tick it to cross it off, untick it, and delete one. When you remove the last task, the empty-state message appears. Every action follows the same shape — update `tasks`, call `render` — and the page stays perfectly in sync with the data. That single discipline is what makes the app reliable.

A couple of notes on the new pieces. `tasks.forEach(function (task, index) { ... })` gives you both the task and its position, and you use `index` to update or splice the right one. Each handler closes over the `index` value from the moment its element was built — a snapshot, not a live reference. That is correct here only because every action re-renders immediately, throwing away the old elements and rebuilding fresh ones with up-to-date indexes; if you reused these elements without re-rendering, the captured indexes could point at the wrong task. `splice` is a new array method here: `splice(index, 1)` removes one item starting at `index` (where `push` and `pop` only added or removed at the end, `splice` can remove from any position). And because the checkbox and delete listeners are attached inside `render`, every freshly drawn task gets its own working handlers.

<ReadMore title="Why frameworks exist: from manual DOM to React and Next.js">
Look back at your `render` function. Every time the data changes, you wipe the list and rebuild every element by hand with `createElement`, `textContent`, `classList`, and fresh event listeners. For a to-do list that is fine — but as an app grows, keeping the DOM and your data in sync this way gets tangled fast, and it is easy to update one and forget the other. Modern frameworks were built to remove exactly this chore. **React** (currently version 19) flips the model: instead of telling the browser *how* to update each element, you write components that **declare** what the UI should look like for a given piece of **state**, and React figures out the changes for you. Under the hood it compares a lightweight in-memory copy of the page — the **virtual DOM** — against the previous one and updates only what actually changed, so you never write the wipe-and-rebuild loop yourself. The mental shift is from "find this element and change it" to "describe the result; let the framework reconcile it." **Next.js** (version 16) is the most popular production framework built on React, adding routing, server rendering, and build tooling on top. The instincts you just built — state in a variable, re-render when it changes, components made of small pieces — are precisely the instincts React formalizes, so the leap will feel natural.
