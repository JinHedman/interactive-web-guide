Your to-do app held its data in an array you typed by hand. Real apps usually get their data from somewhere else â€” a server, over the network. This chapter covers how: what an **API** is, the **JSON** format that data arrives in, and how to request it with **`fetch()`** and render it into the page. The good news is you already know the hard part â€” once the data is an array of objects, you render it exactly the way you rendered tasks in the capstone.

## What an API is

An **API** (Application Programming Interface) is a way for one program to ask another for data or actions. On the web, that usually means a URL you can request that returns data instead of a web page. Hit `https://api.example.com/users` and instead of HTML for a browser to draw, you get back raw data â€” a list of users â€” for *your code* to use however it likes.

You will use a free, public practice API called **JSONPlaceholder**. Visiting `https://jsonplaceholder.typicode.com/users` in your browser returns a list of ten fake users. It needs no key, no login, and allows requests from your own pages â€” perfect for learning.

## JSON: data as text

The data comes back as **JSON** (JavaScript Object Notation). JSON is just text shaped like JavaScript objects and arrays â€” the same braces, brackets, and key/value pairs you already know. That is not a coincidence: JSON was designed so data could travel as text and snap straight back into JavaScript values.

<CodeExample language="js">
{`// A single user from the API, as JSON text:
{
  "id": 1,
  "name": "Leanne Graham",
  "email": "leanne@example.com",
  "company": { "name": "Romaguera-Crona" }
}`}
</CodeExample>

Two small differences from a JavaScript object literal: in JSON every **key is wrapped in double quotes**, and JSON holds only data â€” no functions, no comments. The full `/users` response is an **array** of objects like this one, exactly the array-of-objects shape your to-do tasks used. Once you have that array, rendering it is familiar territory.

## fetch and async/await

You request a URL with the built-in **`fetch()`** function. The catch: a network request takes time â€” milliseconds to seconds â€” and JavaScript does not freeze the page waiting for it. Instead `fetch()` returns a **promise**: a placeholder for a value that will arrive *later*. The clean way to wait for that value is the **`await`** keyword, which pauses inside the function until the result is ready. `await` only works inside a function marked **`async`**.

<CodeExample language="js">
{`async function loadUsers() {
  const response = await fetch("https://jsonplaceholder.typicode.com/users");
  const users = await response.json();
  console.log(users);
}

loadUsers();`}
</CodeExample>

Read it top to bottom, like ordinary step-by-step code â€” that is the whole point of `await`:

1. `await fetch(url)` sends the request and waits for the server to respond. The result is a **`Response`** object â€” the envelope, not the data yet.
2. `await response.json()` reads the response body and parses the JSON text into real JavaScript values. Here that produces an **array of user objects**.
3. `console.log(users)` would print that array in the devtools console.

Note: this is a **network** call, so it will not run in the small preview boxes on this page â€” those are sandboxed and block live requests. You will run it for real in your own browser in the exercise below, where `fetch` works and the console shows the array.

## Render the fetched data

Once `response.json()` hands you an array, you render it with the same loop-and-build pattern from the capstone: clear the container, loop over the array, build one element per item, append. The only new part is *getting* the array; everything after is the DOM work you already know.

<CodeExample language="js">
{`async function loadUsers() {
  const list = document.getElementById("userList");
  const response = await fetch("https://jsonplaceholder.typicode.com/users");
  const users = await response.json();

  list.textContent = "";
  users.forEach(function (user) {
    const li = document.createElement("li");
    li.textContent = user.name + " â€” " + user.email;
    list.append(li);
  });
}

loadUsers();`}
</CodeExample>

Dropped into a page with `<ul id="userList"></ul>`, this fills the list with ten lines, each showing a user's name and email â€” the names and addresses are read straight off each object with dot notation (`user.name`, `user.email`), just like reading any object property. Compare it to the capstone's `render`: same `forEach`, same `createElement`, same `append`. The data now comes from the network instead of a typed-in array, but the rendering is identical.

## When the request fails

Networks are unreliable: the server might be down, or the user offline. A failed `await` throws an error, which you catch by wrapping the calls in **`try...catch`** â€” `try` the request, `catch` any failure and show a message instead of leaving the page blank. (You may also see the older `.catch(...)` attached to a promise; `try...catch` is the natural fit once you are using `async`/`await`.)

<CodeExample language="js">
{`async function loadUsers() {
  const list = document.getElementById("userList");
  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/users");
    const users = await response.json();
    list.textContent = "";
    users.forEach(function (user) {
      const li = document.createElement("li");
      li.textContent = user.name + " â€” " + user.email;
      list.append(li);
    });
  } catch (error) {
    list.textContent = "Could not load users. Please try again.";
  }
}

loadUsers();`}
</CodeExample>

Now a network failure shows a friendly message instead of a silent blank list. That single `try...catch` is enough error handling to start with.

<ReadMore title="Going further: promises and async/await">
You met `async`/`await` here as a recipe; underneath it sits the **promise**, JavaScript's object for a value that arrives later. `await` is really sugar over promises â€” `fetch(url).then(res => res.json()).then(data => ...)` does the same work with `.then()` callbacks, and you will see both styles in real code. The mental model worth keeping: a promise is *pending*, then either *fulfilled* with a value or *rejected* with an error, and `await` (with `try...catch`) lets you write that as plain top-to-bottom steps. Learning the promise model fully will make every future async API â€” timers, file reads, other network libraries â€” click into place.

- [MDN â€” Using promises](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Using_promises)
- [MDN â€” Using the Fetch API](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API/Using_Fetch)
</ReadMore>
