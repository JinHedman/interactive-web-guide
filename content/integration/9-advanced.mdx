This is an **optional, deeper chapter**. The capstone app already works: it adds, completes, and deletes tasks, and the fetch chapter renders live data. But that app has soft spots a real project would harden. It attaches a fresh listener to every element on every render. It forgets everything the moment you reload the page. Its fetch trusts that a response means success. This chapter fixes each of those — not with new frameworks, just with patterns the platform already gives you.

Each section teaches one technique with a runnable example, then you will drill them and finish with a quiz. None of this is required to ship the capstone — it is the next layer of craft.

## Event delegation: one listener for many items

Look back at the capstone. Inside `render`, every task gets its own delete button *and* its own click listener. Re-render ten times and you have created and thrown away ten sets of listeners. It works, but it is wasteful, and it ties the listener's lifetime to the element's.

There is a better way, and it leans on something you already met: **`event.target`**, the element the event actually came from. Events **bubble** — a click on a button inside a `<li>` inside a `<ul>` fires on the button, then the `<li>`, then the `<ul>`, all the way up. So you can put **one** listener on the parent and let every child's clicks bubble up to it. That is **event delegation**.

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
  </head>
  <body>
    <ul id="list">
      <li>Buy milk <button class="del">Delete</button></li>
      <li>Walk the dog <button class="del">Delete</button></li>
      <li>Write code <button class="del">Delete</button></li>
    </ul>
    <script src="script.js"></script>
  </body>
</html>`
  },
  {
    name: "script.js",
    lang: "js",
    code: `const list = document.getElementById("list");

// ONE listener on the parent — not one per button.
list.addEventListener("click", function (event) {
  // Did the click come from a delete button? Ignore everything else.
  if (event.target.matches(".del")) {
    const li = event.target.closest("li");
    li.remove();
  }
});`
  }
]} />

Two new helpers make this clean. `event.target.matches(".del")` asks "is the clicked element a delete button?" — a guard so the listener ignores clicks elsewhere in the list. `event.target.closest("li")` walks **up** from the clicked element to the nearest ancestor that matches the selector — here, the `<li>` the button lives in. That is what you delete.

Why this beats a listener per item: you attach it **once**, and it keeps working for items added *later* — even ones that did not exist when the listener was set up. The capstone's footnote worried that listeners attached inside `render` are fragile and rebuilt constantly; delegation sidesteps the whole problem. One listener, on a parent that outlives every child.

## Persisting state with localStorage

Your capstone's tasks live in a JavaScript array. Reload the page and that array is recreated from scratch — your work is gone. **`localStorage`** is a small key/value store the browser keeps on disk, per site, that survives reloads and even closing the tab.

It only stores **strings**, though. Your state is an array of objects. The bridge is the JSON pair you met in the fetch chapter: **`JSON.stringify`** turns a value into a JSON string to save, and **`JSON.parse`** turns that string back into real values to load.

<CodeExample language="js" preview>
{`const tasks = [
  { text: "Buy milk", done: false },
  { text: "Walk the dog", done: true }
];

// SAVE: object/array -> JSON string -> localStorage
localStorage.setItem("tasks", JSON.stringify(tasks));

// LOAD: localStorage -> JSON string -> object/array
const saved = localStorage.getItem("tasks");
const loaded = JSON.parse(saved);

console.log(loaded[1].text); // "Walk the dog"
console.log(loaded.length);  // 2`}
</CodeExample>

In a real app you do two things: **load** once at startup, and **save** after every change. `getItem` returns `null` when nothing has been saved yet (a first visit), so guard the parse with a fallback so the first load does not crash:

<CodeExample language="js" preview>
{`function loadTasks() {
  const saved = localStorage.getItem("tasks");
  // No saved data yet? Start with an empty list.
  return saved ? JSON.parse(saved) : [];
}

function saveTasks(tasks) {
  localStorage.setItem("tasks", JSON.stringify(tasks));
}

let tasks = loadTasks();
console.log(tasks); // [] on a first visit; your saved array after that

tasks.push({ text: "New task", done: false });
saveTasks(tasks); // call this after every change that should persist`}
</CodeExample>

Slot this into the capstone's rhythm: call `loadTasks()` to fill `tasks` at the top of the script, and call `saveTasks(tasks)` at the end of `render` (or in each handler, right after you change the array). The discipline stays the same — *change the array, then render* — you just add *and save*.

A couple of cautions. `localStorage` is per-origin and visible to anyone at the keyboard, so never store secrets or passwords in it. And a saved string can be corrupt or hand-edited; wrapping the parse in `try...catch` and falling back to `[]` keeps a bad value from breaking the whole app.

## Robust fetch: ok, status, and UI states

The fetch chapter wrapped the request in `try...catch` — good for a dropped connection. But there is a trap: a `404 Not Found` or `500 Server Error` is still a **successful HTTP exchange**, so `fetch` does **not** throw. The promise resolves; `try...catch` never fires. You have to check the response yourself with **`response.ok`** (true for status codes 200–299) and **`response.status`** (the number).

<CodeExample language="js">
{`async function loadUser(id) {
  const response = await fetch("https://jsonplaceholder.typicode.com/users/" + id);

  // fetch does NOT throw on 404/500 — check it yourself.
  if (!response.ok) {
    throw new Error("Request failed with status " + response.status);
  }

  return response.json();
}`}
</CodeExample>

Throwing on a bad status means one `try...catch` now handles *both* a network failure and a server error. With that in place, a data-driven UI really has **four** states to show, not one: loading, success, empty (the request worked but came back with nothing), and error. Handling each is the difference between an app that feels broken and one that feels finished.

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
  </head>
  <body>
    <button id="load">Load users</button>
    <div id="status"></div>
    <ul id="list"></ul>
    <script src="script.js"></script>
  </body>
</html>`
  },
  {
    name: "script.js",
    lang: "js",
    code: `const loadBtn = document.getElementById("load");
const statusEl = document.getElementById("status");
const list = document.getElementById("list");

async function loadUsers() {
  // 1. Loading state
  statusEl.textContent = "Loading...";
  list.textContent = "";

  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/users");
    if (!response.ok) {
      throw new Error("Status " + response.status);
    }
    const users = await response.json();

    // 2. Empty state
    if (users.length === 0) {
      statusEl.textContent = "No users found.";
      return;
    }

    // 3. Success state
    statusEl.textContent = "";
    users.forEach(function (user) {
      const li = document.createElement("li");
      li.textContent = user.name;
      list.append(li);
    });
  } catch (error) {
    // 4. Error state
    statusEl.textContent = "Could not load users. Please try again.";
  }
}

loadBtn.addEventListener("click", loadUsers);`
  }
]} />

This is a real network call, so it runs in your own browser via Live Server, not in the sandboxed preview here. Read the flow: show *Loading* immediately, then either replace it with the list, with an empty message, or with an error message. The user always knows what is happening.

Sending data follows the same shape, with extra arguments. Conceptually, a `POST` request passes a second argument to `fetch` — a `method`, a `headers` object declaring the body is JSON, and a `body` that is your data run through `JSON.stringify`:

<CodeExample language="js">
{`// Conceptual: sending data with POST (needs a real server to receive it).
async function addUser(user) {
  const response = await fetch("https://jsonplaceholder.typicode.com/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(user)
  });
  return response.json();
}`}
</CodeExample>

Notice the same `JSON.stringify` you used for `localStorage` — turning a JavaScript object into a string is the universal move whenever data leaves your program, whether to disk or across the network. Reading (`GET`) is the default; writing adds the `method`/`headers`/`body` trio.

## Keyboard events and cleanup

Click and submit are not the only events. **`keydown`** fires when a key goes down, and the event object carries **`event.key`** — a readable name like `"Enter"`, `"Escape"`, or `"a"`. This is how you wire up keyboard shortcuts: Enter to confirm, Escape to cancel.

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
  </head>
  <body>
    <input id="field" placeholder="Type, then Enter or Escape">
    <p id="out">Waiting...</p>
    <script src="script.js"></script>
  </body>
</html>`
  },
  {
    name: "script.js",
    lang: "js",
    code: `const field = document.getElementById("field");
const out = document.getElementById("out");

field.addEventListener("keydown", function (event) {
  if (event.key === "Enter") {
    out.textContent = "Submitted: " + field.value;
  } else if (event.key === "Escape") {
    field.value = "";
    out.textContent = "Cancelled.";
  }
});`
  }
]} />

Type in the box, then press Enter to "submit" or Escape to clear. Reading `event.key` and branching on its value is the whole pattern — no special key-code numbers to memorize.

The flip side of `addEventListener` is **`removeEventListener`**: it detaches a listener so it stops firing. To remove one you must pass the **exact same function reference** you added — which means the handler has to be a *named* function, not an inline anonymous one. This matters for temporary listeners, like a global Escape handler that should only exist while a dialog is open.

<CodeExample language="js" preview>
{`// A named handler so we can remove it later by reference.
function onKey(event) {
  if (event.key === "Escape") {
    console.log("Escape pressed — closing.");
    // Clean up: stop listening once the job is done.
    document.removeEventListener("keydown", onKey);
  }
}

// Start listening (e.g. when a dialog opens).
document.addEventListener("keydown", onKey);

// An anonymous function could NOT be removed — there is no
// reference to pass back to removeEventListener.`}
</CodeExample>

Removing listeners you no longer need prevents stray handlers from firing and keeps memory from leaking in long-running apps. Pair every "open" that adds a temporary listener with a "close" that removes it.

## Rendering a list with map and join

You have built lists element-by-element with `createElement`. Once you know `map` (from the JS chapters), there is a terser declarative style: turn each data item into an HTML **string**, `join` them into one string, and assign it to the parent's `innerHTML` in a single shot.

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
  </head>
  <body>
    <ul id="list"></ul>
    <script src="script.js"></script>
  </body>
</html>`
  },
  {
    name: "script.js",
    lang: "js",
    code: `const list = document.getElementById("list");

const fruits = ["Apple", "Banana", "Cherry"];

// Map each item to an <li> string, join into one string, set once.
list.innerHTML = fruits
  .map(function (fruit) {
    return "<li>" + fruit + "</li>";
  })
  .join("");`
  }
]} />

This is concise and reads like the JSX you will meet in React. But there is a sharp edge: `innerHTML` **parses its string as HTML**. If any piece of that string came from a user (a typed task, a name from a form), a malicious value like `<img src=x onerror="...">` would run its script the moment the browser parses it — a cross-site scripting (XSS) hole. So the rule:

- **Safe:** the data is fixed values you control, as above. Use `map().join("")` freely.
- **Not safe:** any part comes from user input or a network response. Build nodes with `createElement` and set text with `textContent`, which treats input as plain text and never as markup.

When in doubt, prefer `textContent` and `createElement`. The `map().join("")` shortcut is a fine tool for trusted, fixed data; it is not a license to drop your guard around input.

<ReadMore title="How frameworks handle this for you">
Every technique here is something modern frameworks fold into the model so you rarely write it by hand. In **React**, you do not attach listeners to each rendered item or refactor to delegation — you write `onClick` on the element and React manages the underlying listeners efficiently for you. You do not manually `createElement` and set `innerHTML`; you return markup-like **JSX** (`<li>{item.text}</li>`) and React escapes interpolated values by default, which closes the XSS hole the `map().join("")` section warned about. The four fetch states — loading, success, empty, error — become a tiny piece of state you track and render conditionally, and libraries like **TanStack Query** or the framework's data layer give you those states for free. The skills stay valuable: knowing what delegation, persistence, and `response.ok` actually do is exactly what lets you reason about a framework when its abstraction leaks.
