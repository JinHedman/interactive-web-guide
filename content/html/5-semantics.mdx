<InShort>Semantic elements like `<header>`, `<nav>`, `<main>`, `<section>`, `<article>`, and `<footer>` name the **role** of each region; fall back to `<div>`/`<span>` only when none fits.</InShort>

You could build any page out of plain boxes, but HTML offers elements that name the *role* of each region — header, navigation, main content, footer. These are called semantic elements, and using them makes your page easier for assistive tech, search engines, and your future self to understand. In this chapter you reorganize your profile page around semantic landmarks.

<WhyItMatters>
A `<div class="header">` and a `<header>` can look identical on screen, so it is tempting to ask why the distinction matters. It matters to everyone who is not looking at the pixels: a screen-reader user jumps straight to "main content," a search engine reads your structure to rank the page, and the next developer understands each region at a glance. Same visual result, far more meaning.
</WhyItMatters>

## Landmarks name the regions of a page

Semantic landmark elements describe what a region *is*, not how it looks. The common ones:

- `<header>` — introductory content at the top, often a logo or page title.
- `<nav>` — a block of navigation links.
- `<main>` — the page's primary content. Use exactly one per page.
- `<section>` — a thematic grouping of content, usually with its own heading.
- `<article>` — a self-contained piece that would make sense on its own, like a blog post or a card.
- `<footer>` — closing content at the bottom, like contact info or copyright.

<CodeExample language="html" preview>
{`<header>
  <h1>Ada Lovelace</h1>
  <nav>
    <a href="#about">About</a> ·
    <a href="#projects">Projects</a>
  </nav>
</header>

<main>
  <section id="about">
    <h2>About me</h2>
    <p>A short bio.</p>
  </section>

  <section id="projects">
    <h2>Projects</h2>
    <article>
      <h3>Analytical Engine notes</h3>
      <p>My write-up on an early computing idea.</p>
    </article>
  </section>
</main>

<footer>
  <p>Contact: ada@example.com</p>
</footer>`}
</CodeExample>

Notice the page reads like an outline of itself. A screen-reader user can jump straight to "main" or list all the navigation regions, because these elements announce their roles.

## section vs article

The two grouping elements confuse people. A quick guide:

- Use `<section>` for a themed part of a larger whole — a chunk of the page that has a heading.
- Use `<article>` when the content would stand on its own if you lifted it out — a single project card, a comment, a news story.

When in doubt, ask: would this still make sense pasted somewhere else by itself? If yes, it is probably an `<article>`.

## div and span: the fallbacks

Sometimes you need to group elements purely for styling or scripting, with no semantic meaning. That is what `<div>` (a block-level generic box) and `<span>` (an inline generic box) are for.

<CodeExample language="html">
{`<!-- Generic grouping for layout, no inherent meaning -->
<div class="card">
  <p>I am <span class="highlight">currently available</span> for projects.</p>
</div>`}
</CodeExample>

Reach for `<div>` and `<span>` only when no semantic element fits. If a region is really the navigation, use `<nav>`, not `<div class="nav">`.

## Block vs inline

Elements fall into two display categories by default:

- **Block** elements start on a new line and stretch across the available width. Examples: `<p>`, `<h1>`, `<div>`, `<section>`.
- **Inline** elements sit within a line of text and take only as much width as their content. Examples: `<a>`, `<strong>`, `<em>`, `<span>`.

<CodeExample language="html" preview>
{`<p>This paragraph is a <strong>block</strong> element.</p>
<p>The <em>emphasized words</em> here are <a href="#">inline</a> elements.</p>`}
</CodeExample>

This is why `<div>` (block) stacks vertically while `<span>` (inline) flows inside text. You will control display in detail with CSS later; for now, just know the two categories exist.

## Why semantics matter

Semantic markup pays off in three ways:

- **Accessibility** — screen readers expose landmarks so users can skip directly to navigation or main content.
- **SEO** — search engines read your structure to understand and rank the page.
- **Maintainability** — `<nav>` and `<footer>` tell the next developer (often future you) what each region is at a glance.

Same visual result, far more meaning. That trade is almost always worth it.
