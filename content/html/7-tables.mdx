A table presents data in rows and columns — a schedule, a price list, a set of stats. HTML tables have a specific structure of nested elements, and getting the headers right is what makes them readable and accessible. In this chapter you add a small data table to your profile page, such as your skills with proficiency levels.

## Rows and cells

A table is built from rows, and each row holds cells. The elements:

- `<table>` wraps the whole thing.
- `<tr>` is a table row.
- `<th>` is a header cell (bold and centered by default).
- `<td>` is a data cell.

<CodeExample language="html" preview>
{`<table>
  <tr>
    <th>Skill</th>
    <th>Level</th>
  </tr>
  <tr>
    <td>HTML</td>
    <td>Confident</td>
  </tr>
  <tr>
    <td>CSS</td>
    <td>Learning</td>
  </tr>
</table>`}
</CodeExample>

Each `<tr>` is a row read left to right; the cells inside it line up into columns. Use `<th>` for the labels at the top of each column so the table announces them as headers.

## Group rows with thead and tbody

For a clearer structure, group the header row in a `<thead>` and the data rows in a `<tbody>`. This separates the labels from the data, both visually for tools and semantically for screen readers.

<CodeExample language="html" preview>
{`<table>
  <thead>
    <tr>
      <th>Skill</th>
      <th>Level</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>HTML</td>
      <td>Confident</td>
    </tr>
    <tr>
      <td>JavaScript</td>
      <td>Beginner</td>
    </tr>
  </tbody>
</table>`}
</CodeExample>

`<thead>` holds the header row; `<tbody>` holds the body rows. On long tables this also lets the header stay put while the body scrolls.

## Add a caption

A `<caption>` gives the table a title, placed as the first child of `<table>`. Screen readers announce it, so the user knows what the table is about before reading the cells.

<CodeExample language="html" preview>
{`<table>
  <caption>My skills and current level</caption>
  <thead>
    <tr><th>Skill</th><th>Level</th></tr>
  </thead>
  <tbody>
    <tr><td>HTML</td><td>Confident</td></tr>
    <tr><td>CSS</td><td>Learning</td></tr>
  </tbody>
</table>`}
</CodeExample>

## Tables are for data, not layout

In the early web, people used tables to position things on a page. Do not. Tables are for **tabular data** — information that genuinely belongs in rows and columns. For arranging boxes on a page, use CSS layout (the CSS module covers this). Misusing tables for layout confuses screen readers, which try to read out rows and columns that are not really data.

The test: if your content has a natural row-and-column relationship, a table is right. If you just want things side by side, it is a layout job for CSS.
