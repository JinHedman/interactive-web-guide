HTML gives a page structure. CSS gives it style. JavaScript gives it *behavior* — the page can respond to clicks, change its own content, do math, and talk to servers. This chapter is about getting JavaScript to run at all: where the code goes, how to see its output, and how to leave notes for yourself.

## What JavaScript adds

HTML and CSS describe a page once, when it loads. JavaScript runs *after* that and can keep changing things. A countdown timer, a form that checks your input as you type, a button that adds a row to a list — all behavior, all JavaScript.

For now we will not touch the page itself. We will write small programs and read their output in the **console**, a panel built into every browser for developers.

## Writing to the console

`console.log` prints a value to the console. It is the simplest way to see what your code is doing.

<CodeExample language="js" preview>
{`console.log("Hello from JavaScript");
console.log("This runs in the browser console.");`}
</CodeExample>

Each `console.log(...)` prints one line. You can pass it text, numbers, or the result of a calculation.

<CodeExample language="js" preview>
{`console.log("Two plus two is:");
console.log(2 + 2);`}
</CodeExample>

In these lessons, the result pane on the right shows whatever you `console.log`. In a real browser you would open the devtools console to see the same output (more on that below).

## Adding a script to a page

In a real project, JavaScript lives in its **own `.js` file**, separate from your HTML. You attach it to the page with a `<script>` tag and a `src` attribute, just like a stylesheet. These are two files in the same folder:

<CodeExample preview files={[
  {
    name: "index.html",
    lang: "html",
    code: `<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <title>My page</title>
  </head>
  <body>
    <h1 id="greeting">Loading...</h1>

    <script src="script.js"></script>
  </body>
</html>`
  },
  {
    name: "script.js",
    lang: "js",
    code: `document.getElementById("greeting").textContent = "Hello from a separate file!";`
  }
]} />

Notice where the `<script>` sits: at the very bottom of `<body>`, just before `</body>`. The browser reads the page top to bottom. If the script runs before the elements above it exist, it cannot find them. Putting the script last means the whole page is ready by the time your code runs.

The `src="script.js"` attribute is the link between the two files: it tells the browser to load and run that file at this point in the page. Keeping HTML and JavaScript in separate files is the normal way to work — the markup describes the page, the script describes its behavior.

Do not worry about that `document.getElementById` line yet — that is the DOM, which the final chapter of this module introduces. For now, just notice that the script in `script.js` changed the heading after the page loaded.

## Opening the devtools console

On your own machine, you read `console.log` output in the browser's **devtools console**:

- **Chrome / Edge**: press `F12`, or right-click the page and choose *Inspect*, then click the **Console** tab.
- **Firefox**: press `F12`, then the **Console** tab.
- **Safari**: enable the Develop menu in Settings, then press `Option` + `Cmd` + `C`.

Keep the console open while you work. Errors show up there too, in red — reading them is half of debugging.

## Comments

A comment is a note in your code that JavaScript ignores. Use it to explain *why* something is there.

<CodeExample language="js" preview>
{`// This is a single-line comment. JavaScript skips it.
console.log("Comments do not run.");

/*
  This is a multi-line comment.
  Useful for longer explanations.
*/
console.log("But this line does run.");`}
</CodeExample>

`//` comments out the rest of a line. `/* ... */` comments out everything between the markers, across multiple lines.
